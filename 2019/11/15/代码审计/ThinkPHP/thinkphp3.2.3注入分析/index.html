ThinkPHP3.2.3注入分析

前言

近段时间一直在学习tp3，终于搞清楚运作流程，也顺便分析下tp3.2.3爆出的注入漏洞。

环境搭建

利用git下载漏洞修复之前的版本

下载源码： git clone https://github.com/top-think/thinkphp.git

使用git checkout 命令将版本回退到上一次commit：

漏洞分析

find()和select()

find()和select()函数处理流程相似，差别在于find存在limit 1限制，仅返回一条记录

本部分以find()为例

审计复现

payload

table

函数调用流程

进入到public function find($options = array())方法内

文件位置：ThinkPHP/Library/Think/Model.class.php

payload中参数id是以数组形式传入，则函数体内$options=array('table'=>'user where........')

首先经过第一次if判断，但此处由于$options是以数组形式，因此绕过

$pk的值是当前表的主键信息：

若当前表仅有一个主键，则$pk的值为string

若当前表有多个主键，这$pk的值是以数组的形式存在

该值在本漏洞中非常重要，当$pk为string类型时，会导致接下来的部分判断绕过；否则payload不生效

接下来的语句块是根据复合主键查询，因此处表为单主键，$pk为字符串类型，因此绕过判断

进入到_parseOptions($options)函数体中

$options可控，同时本次payload中$options仅存在table字段，不存在where字段，可顺利通过函数验证

往下进入到select()方法中

再进入到select方法体中buildSelectSql()

进入到parseSql()中，该方法主要用来利于前面获取的各字段信息，拼接SQL语句。

在此方法中，直接取出$options[‘table’]的值作为查询语句中的table

因而SQL语句为

alias

传入第二个payload

?id[alias]=where%201%20and%20updatexml(1,concat(0x7e,user(),0x7e),1)–

alias指别名，即SQL语句中表的别名

主要漏洞在于_parseOptions()方法中 ：

直接将别名拼凑在表名的后面，从而导致了注入

最终注入语句：

where

跟进到select()方法中

进入buildSelectSql()方法中

parseSql()中拼接时将$option[‘where’]作为了where部分，拼接后导致注入

小试牛刀

根据上面的漏洞复现发现，上面漏洞都有一个共同点：涉及parseSql()方法且$option可控，那么其他字段是否也存在注入呢？

答案是肯定的……

group

payload

获取$options[‘’group]中的值是parseGroup()完成的，因$options可控，且没有任何过滤，因此导致注入

理论上其他字段都存在相应的漏洞风险。

输入payload后跟进到delete()函数

首先进入到_parseOptions($optinos)

该函数非常重要，在本例中没有起到作用，顺利通过

回到delete()函数中，进入$result = $this->db->delete($options);中

进入parseWhere()中，$where为字符串，直接赋值给$whereStr

拼接到SQL语句中

注入成功

table

payload：

http://127.0.0.1/tp3_2/Home/Index/sql?id[table]=user where 1 and updatexml(1,concat(0x7e,user(),0x7e),1)–&id[where]=1

进入到$this->db->delete($options);中

跟进到parseTable($options['table'])，显然传入参数为字符串

在本方法中会先对传入参数值进行分割，再合并成数组，这里不会对payload造成影响

回到delete()方法中，成功拼接成注入语句

这里的payload中必须传入id[where]，原因在于delete()方法中，若无where字段值，直接返回false

漏洞修复

根据官方的修复来看，主要是在各个方法中不再使用$options，改用$this->options，且不再使用_parseOptions($options)，因此$options不再可控，漏洞被修复。

parseWhereItem()造成的注入

原本认为tp3中数据查询相关操作如果使用数组为参数会非常安全，但发现以下类型的漏洞

update()注入

漏洞复现

控制器

payload

传入payload后，首先进入到Include/Library/Think/Model.class.php中where()方法中，在where方法中确定SQL语句中where条件部分的键数

随后进入到Include/Library/Think/Model.class.php中的save()

通过前面的数据处理解析服务端数据库中的数据字段信息，字段数据类型，再到_parseOptions表达式分析，获取到表名，数据表别名，记录操作的模型名称，再去调用回调函数进入update()

Include/Library/Think/Db/Driver.class.php中update()

首先经过parseBind和parseSet()生成$bind变量，以及拼接了部分SQL语句

随后进入到parseWhere()方法中，关键在于其函数体中parseWhereItem()方法，在下面代码块中14行处，将$exp拼接在$wherestr后

再次回到Include/Library/Think/Db/Driver.class.php中update()中

进入到execute()中，下面对SQL中:0进行替换。

此处$this->bind变量的值是：

这是为什么payload中使用的是”0”而不是其他数字的原因，否则将无法替换SQL语句中的:X

find()注入

漏洞复现

复现要求：think_filter()中未过滤exp

payload

具体分析同上面update注入流程相同

不使用bind参数是因为find和select中不会对:X进行替换，造成语法错误

漏洞修复

漏洞造成的一个原因便是Include/Common/functions.php中think_filter中忽略了对bind参数的过滤

修补之后，系统会在传入值bind后添加一个空格，导致parseWhereItem()中无法将传入的payload拼接到where中

小结

个人感觉thinkphp3.2.3爆出的SQL漏洞都比较鸡肋，即使不修复以上漏洞，开发者仍然可以规避上面的注入风险（经实例审计后发现的~）
